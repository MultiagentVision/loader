# BFF vs BFF-mock vs Frontend expectations (for integration discussion)

| Module | Frontend (`frontendChessAI`) calls | `bff-mock` (Express) | Real `bff` (FastAPI) | Frontend expects | Integration gap / questions |
|---|---|---|---|---|---|
| Base URL / path prefix | FE uses `axios.create({ baseURL: import.meta.env.VITE_API_URL })` then calls paths like `POST /auth/login`, `GET /tournaments/...` (so `VITE_API_URL` must already include `/api/v1` in production) | `app.use("/api/v1", routes)` => `/api/v1/auth/login`, `/api/v1/tournaments/...` | OpenAPI uses `/api/v1/...` | Correct URL building (no extra `/api/v1` duplication) | Confirm what `VITE_API_URL` is set to in prod/stage (does it end with `/api/v1`?) |
| Auth: login/refresh/logout | `POST /auth/login`<br>`POST /auth/refresh` body `{ refreshToken }`<br>`POST /auth/logout` body `{ refreshToken }` | Has `/api/v1/auth/login`, `/api/v1/auth/refresh`, `/api/v1/auth/logout` | OpenAPI: `/api/v1/auth/login`, `/api/v1/auth/refresh`, `/api/v1/auth/logout` | `tokens.accessToken` and `tokens.refreshToken` + user profile (login) | Contract seems mostly aligned; still confirm token field casing and exact error shapes for 401/400 |
| User profile | `GET /user/profile` | `/api/v1/user/profile` | `/api/v1/user/profile` | `UserProfileDto` | Mostly aligned |
| Tournaments (watch list / details / game) | List:<br>`GET /tournaments?cursor&pageSize&participantId`<br>Details:<br>`GET /tournaments/:tournamentId`<br>Game:<br>`GET /tournaments/:tournamentId/:gameId` | List is cursor-based:<br>`GET /api/v1/tournaments?cursor&pageSize`<br>Response: `{ items, meta: { nextCursor, hasNext, pageSize, totalCount }}` | OpenAPI + routers: page-based + status filter:<br>`GET /api/v1/tournaments?page&pageSize&status`<br>Response model: `TournamentListResponse { items, meta { page, pageSize, totalCount, hasNext } }` | FE watch UI uses `cursor` and expects `meta.hasNext`, `meta.pageSize`, and it uses `cursor` for infinite scroll concatenation | Main mismatch: FE uses `cursor` and `participantId`; real `bff` uses `page` and `status`. Ask the backend: will real `bff` support `cursor/participantId`, or will FE switch to `page/status`? |
| Notifications / subscriptions (alerts vs notifications) | `alertsApi.ts` calls:<br>`GET /alerts/subscriptions`<br>`POST /alerts/subscriptions` body `{ subscriptions: [{ source, channels: { push } }] }`<br>`DELETE /alerts/subscriptions/:source` | `bff-mock` currently does NOT expose notification/alerts routes in `src/routes/index.ts` (only auth/user/tournaments/recognition/streams) | OpenAPI provides notifications:<br>`/api/v1/notifications/subscriptions`<br>`/api/v1/notifications/subscriptions/{source}`<br>and push-subscriptions endpoints | FE expects `AlertSubscriptionsResponse` shape:<br>`{ success, data: { subscriptions: [{ source, channels: { push } }] } }` | Critical mismatches to discuss with backend:<br>1) Path: FE uses `/alerts/*`, real `bff` uses `/notifications/*`.<br>2) Request schema: real `bff` expects `channels.email` AND `channels.push` (email is required), but FE sends only `channels.push`. |
| Notifications / subscriptions: what backend expects in payload | FE sends: `{ source, channels: { push } }` | (Not implemented in mock yet) | `notifications.py` expects `channels.email` and `channels.push` | Backend should either accept the FE payload or FE should be updated | Ask backend to add `alerts/*` aliases and/or make `channels.email` optional with default `false` |
